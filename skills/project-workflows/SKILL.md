# Project Workflows

把「新开 / 维护 / 学习」三种工作方式，收敛成可执行的输出契约与路由规则。技术选型横切仍用 `tech-preferences`；系统设计取舍用 `unix-software-design`。

## 工作模式（选一条主 lane）

| Lane | 含义 |
|------|------|
| **新开**（new-project） | 从模糊想法到最小可行起点，先证明最大不确定性 |
| **维护**（maintain-project） | 在现状约束下选最值得推进的一步并做实 |
| **学习**（learn-project） | 读他人项目，产出可迁移模式，不盲目照搬 |

- 保持当前模式在推理与交接中**显式**。
- 一次只选一个主模式；若用户混合需求，**按顺序拆**（例如先学习再维护），不要糊成一段泛谈。
- 同一模式内、彼此独立的子问题可**并行**；跨模式时优先在一个模式内并行，再切下一模式。
- 非琐碎任务时，拆解应写明：子问题、依赖、**最低可交付物（output floor）**、哪些可并行。
- 不要把明显该交给专项能力的活硬塞进总控，除非任务太小或紧耦合合成成本过高。
- 向用户交付前，应有可合并进仓库的**具体产出**（而非只有意向）。

## 路由（模式不清或混合时）

- 识别混合请求时，先列出包含的模式，再约定顺序或并行边界。
- 期望：混合输出里**明确分段**（学习段 / 维护段等），不混淆为单一含糊建议。

## 三种 Packet（输出契约）

### Kickoff packet（新开）

至少包含：目标、约束、最小可行形态、第一阶段要证明的点、下一步 **3 个动作**。

**工作方式：** 先明确目标、约束、成功标准；涉及选型先加载 `tech-preferences`；压缩成最小可行形态；优先证伪/证实**最大不确定性**，而非铺完整长期架构。若用户未给具体方向，可用「通用小型项目」示例产出，并提示用户替换为自己的想法。

**避免：** 第一版堆过多抽象；无证据时设计整套长期架构；一长串未筛选技术选项。

### Continuation packet（维护）

至少包含：当前状态、这次选定推进的点、为什么先做这个、具体下一步改动、风险与后续事项。

**工作方式：** 从当前代码与目录出发，不从理想蓝图出发；识别现状、卡点、最接近完成的路径；缩小到一个改动面；除非当前方案明显不行，否则**延续已有模式**，不轻易重开选型。

**避免：** 无证据建议全量重写；一次开太多战线；无视 repo 已有约定。

### Learning packet（学习）

至少包含：项目快照、值得借鉴的模式、不宜直接照搬的部分、可迁移到用户项目的具体建议。

**工作方式：** 先定学习目标；抓切面（目录、核心抽象、边界、工具链、演进痕迹）；区分「值得借 / 不要硬抄 / 值得再挖」；若与用户当前项目相关，写明**迁到哪里**。

**避免：** 只有宏观夸奖；忽略原项目上下文；把「别人这样做」等同于「你也该这样」。

## 跨切

- 有意义的技术选择：先 `tech-preferences`；偏好发现也是工作的一部分，能从请求与 repo 推断就少问。
- 代码注释、docstring、README 增补、设计笔记默认 **英文**，除非用户明确要求其他语言；**对话语言**与用户一致。
- 不虚构「全局记忆」职责；本 skill 管流程与边界，不管持久记忆文件。

## 与各 lane 的「不适用」

- **新开** 不适用：纯日常推进已有仓库（用维护）；主要目标是读别人项目（用学习）。
- **维护** 不适用：从零开 repo/原型（用新开）；主要目标是提炼他人经验（用学习）。
- **学习** 不适用：只推进自己项目日常（用维护）；只起新项目（用新开）。
